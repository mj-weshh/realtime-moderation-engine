# Product Requirements Document (PRD): Real-Time Toxicity & Misinformation Streamer

## 1. Executive Summary

The Real-Time Toxicity and Misinformation Streamer is a high-throughput, distributed data pipeline designed to ingest, analyze, and visualize simulated social media traffic. By leveraging a fine-tuned transformer model on the `google/civil_comments` dataset, the system scores incoming text streams for multiple dimensions of toxicity and bias in real time. It utilizes Apache Kafka for event streaming, Neo4j for mapping malicious network clusters, and Next.js with WebSockets for a live, high-performance command-center interface.

This project demonstrates production-grade Machine Learning Operations (MLOps), distributed systems architecture, and strict adherence to SOLID software engineering principles.

---

## 2. System Architecture & Tech Stack

The system is composed of five decoupled, containerized microservices.

### Tech Stack

* **Simulated Data Producer:** Python
* **Message Broker:** Apache Kafka (with Zookeeper/KRaft)
* **ML Inference Consumer:** Python, PyTorch, Hugging Face `transformers`
* **Graph Database:** Neo4j
* **WebSocket API & Client UI:** Node.js backend, Next.js, Tailwind CSS, WebSockets

### High-Level Data Flow

1. **Producer:** Reads from a local subset of the `civil_comments` dataset and publishes raw comment payloads to Kafka at a configurable rate (e.g., 50-100 messages/sec).
2. **Kafka Broker:** Receives and queues messages on the `raw_comments` topic.
3. **ML Consumer:** Polls `raw_comments`, runs the text through the fine-tuned BERT model, appends toxicity scores, and does two things simultaneously:
   * Writes the relationship and scores to Neo4j.
   * Publishes the scored payload to a `scored_comments` Kafka topic.
4. **WebSocket Server:** Subscribes to `scored_comments` and broadcasts high-severity alerts and graph updates to connected frontend clients.
5. **Next.js Client:** Renders the live data stream, highlighting toxic clusters using a dark-mode, cinematic, and clean visual aesthetic.

## 3. Data Models & Contracts

### 3.1 Source Dataset

* **Source:** Hugging Face `google/civil_comments`
* **Features Used:** `text`
* **Target Labels:** `toxicity`, `severe_toxicity`, `obscene`, `threat`, `insult`, `identity_attack`, `sexual_explicit` (All floats: 0.0 to 1.0)

### 3.2 Kafka Event Payloads

**Topic: `raw_comments`**

```json
{
  "event_id": "uuid-1234",
  "user_id": "user-789",
  "text": "The actual comment text goes here.",
  "timestamp": "2026-07-03T04:19:00Z",
  "reply_to_id": "uuid-0000"
}
```

(Note: user_id and reply_to_id will be synthetically generated by the producer to simulate a social network graph, as the original dataset lacks relational metadata).

**Topic: `scored_comments`**

```json
{
  "event_id": "uuid-1234",
  "user_id": "user-789",
  "text": "The actual comment text goes here.",
  "timestamp": "2026-07-03T04:19:00Z",
  "reply_to_id": "uuid-0000",
  "scores": {
    "toxicity": 0.92,
    "identity_attack": 0.85,
    "threat": 0.01
  },
  "is_flagged": true
}
```

### 3.3 Neo4j Graph Schema

**Nodes:**

* User (Properties: user_id)
* Comment (Properties: event_id, text, timestamp, toxicity_score)

**Edges (Relationships):**

* `(User)-[:POSTED]->(Comment)`
* `(Comment)-[:REPLIES_TO]->(Comment)`

## 4. Functional Requirements

### 4.1 Data Producer Service

* Must load the test split of the `civil_comments` dataset into memory.
* Must synthetically assign user IDs and reply chains to create a realistic conversation graph.
* Must allow rate-limiting configuration (e.g., via environment variables) to control throughput.

### 4.2 ML Inference Consumer

* Must load a pre-trained/fine-tuned Hugging Face model (e.g., DistilBERT for lower latency) optimized for CPU or accessible GPU inference.
* Must implement a batching mechanism. Instead of processing one message at a time, it should pull batches (e.g., 16 or 32 messages) from Kafka, perform inference, and output the results to maximize throughput.
* Must commit offsets to Kafka only after successful write to Neo4j and the `scored_comments` topic to prevent data loss.

### 4.3 WebSocket / Backend Service

* Must act as a Kafka Consumer for the `scored_comments` topic.
* Must maintain a pool of active WebSocket connections to the Next.js frontend.
* Should filter out benign comments (e.g., toxicity < 0.5) to save bandwidth, only broadcasting flagged events to the UI.

### 4.4 Next.js Frontend

* Must connect to the WebSocket server on component mount.
* Must render a live, scrolling feed of flagged comments.
* Must render a dynamic force-directed graph (e.g., using `react-force-graph` or D3.js) visualizing the nodes and edges of toxic conversations as they happen.
* Visual design must be high-resolution, modern, and utilize a dark/cinematic palette suitable for a security operations center.

## 5. Non-Functional Requirements

* **Latency:** The time from Producer publishing a message to the UI rendering the flagged alert must be under 2 seconds.
* **Fault Tolerance:** If the ML Consumer crashes, it must resume from the last committed Kafka offset upon restart.
* **Maintainability:** All Python code must be strictly typed, fully documented, and follow SOLID principles (e.g., separating Kafka ingestion logic from PyTorch inference logic).

## 6. Infrastructure & Deployment

The entire system must be orchestratable via a single `docker-compose.yml` file.

**Containers required:**

* zookeeper (or Kafka KRaft mode)
* kafka
* neo4j
* producer_service
* ml_inference_service
* websocket_api
* nextjs_client

Ensure appropriate volume mapping for Neo4j data persistence and model weight caching to avoid re-downloading the Hugging Face model on every container restart.

## 7. Out of Scope for MVP

* Real-time model retraining (Online Learning).
* User authentication and authorization for the Next.js dashboard.
* Cloud deployment scripts (Kubernetes/Terraform). The focus remains on a flawless local Dockerized demonstration.
